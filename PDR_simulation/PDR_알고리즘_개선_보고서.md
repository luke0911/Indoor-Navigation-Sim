# PDR 보폭 추정 알고리즘 개선 보고서

---

## 1. 보폭 계산 파이프라인 (공통 구조)

```
[센서 입력]                    [주파수 경로]                    [진폭 경로]
 globalZ ──→ MA(5) ──→ peak/valley 추출 ──→ ampZ = peak - valley
                              │                                    │
                       걸음 간격(ms)                          A^(1/4) 변환
                              │                                    │
                        freq = 1/간격                    threshold correction
                              │                           (1.441 초과분 30% 보존)
                              ↓                                    │
                           k(f)                              correctedRoot
                     (이중 시그모이드)                              │
                              │                                    │
                              └──────────→ L₀ = k(f) × corrRoot × scale ←──┘
                                                    │
                                          clamp(0.25m ~ 1.40m)
                                                    │
                                                최종 보폭(m)
```

**각 단계 역할**:

- **A^(1/4)**: 진폭 범위를 압축. ampZ 2~10 → 1.19~1.78로 변환하여 큰 진폭이 보폭을 폭주시키는 것을 방지
- **threshold correction**: A^(1/4) > 1.441 초과분의 70%를 깎음. 달리기 등 극단적 진폭 추가 억제
- **k(f)**: 주파수에 따른 보폭 계수. 느린 걸음(0.317) ~ 빠른 걸음(0.700) 범위
- **scale**: 여성 보정 시 0.90 적용 (10% 감소)

## 2. Old → New 변경점

### 2-1. 주파수 계산 방식 변경

**Old: 타임스탬프 간격(Interval) 방식**

```
freq = 1000 / (현재 걸음 시각 - 이전 걸음 시각)
```

- 시스템 타이머 기준으로 두 isStep() 호출 간 시간차 사용
- 문제: 센서 콜백 지연, OS 스케줄링 등으로 **실제 걸음 타이밍과 무관한 노이즈** 혼입

**New: Peak-to-Peak 방식 + 3-sample 이동평균**

```
duration = 현재 up-peak 시각 - 이전 up-peak 시각   (가속도 파형 기준)
smoothed_duration = MovingAverage(3).push(duration)
freq = 1000 / smoothed_duration
```

- 가속도 파형에서 실제 **발이 지면을 밟는 순간**(up-peak)의 시간 차이를 직접 측정
- 3-sample 이동평균으로 추가 안정화

**Interval 대신 Peak-to-Peak를 쓰는 이유**:

- Interval: isStep()이 "걸음 완료"를 판정하는 시점 기준 → 판정 지연이 주파수에 직접 영향
- Peak-to-Peak: 가속도 파형의 **물리적 주기**를 직접 측정 → 판정 로직과 독립적
- 결과: 같은 보행 데이터에서 주파수 변동이 절반 이하로 감소

### 2-2. 출력 스무딩 변경

| | Old | New |
|---|---|---|
| 방식 | 2-window MA + floor(소수 2자리 내림) | 스무딩 없이 raw 출력 |
| 예시 | 0.678m → floor → **0.67m** | 0.678m → **0.678m** |
| 문제점 | floor로 인해 항상 과소추정 (평균 0.5cm) | 없음 |

New는 주파수가 이미 안정화되어 있으므로 출력단 스무딩이 불필요.

### 2-3. 걸음 감지 민감도

| 파라미터 | Old | New | 효과 |
|----------|-----|-----|------|
| 정지 판정 표준편차 | 0.30 | 0.25 | 약한 걸음도 감지 (정지 판정 완화) |
| 적응형 threshold 바닥 | 1.2 | 1.0 | 작은 진폭 걸음 감지 가능 |
| 최소 걸음 간격 | 400ms | 250ms | 빠른 보행(달리기 전환) 대응 |
| 걸음완료 판정 | 진폭 기반 (dynamicUpThresh × 0.5) | 시간 가드 (350ms) + 영점 복귀 | 안정적 완료 감지 |

## 3. Freq CV (주파수 변동 계수)란

**Freq CV = 주파수 표준편차 / 주파수 평균**

- 값이 **0에 가까울수록** 걸음마다 주파수가 일정 → 보폭 추정이 안정적
- 값이 **클수록** 걸음마다 주파수가 들쭉날쭉 → 보폭이 걸음마다 출렁임

예시: 평균 주파수 1.7Hz인 보행에서

- CV = 0.05 → 표준편차 0.085Hz → 대부분 1.6~1.8Hz 내 → **안정**
- CV = 0.15 → 표준편차 0.255Hz → 1.45~1.95Hz 산포 → **불안정**

보폭은 k(f)를 통해 주파수에 의존하므로, 주파수 CV가 높으면 **같은 속도로 걸어도 보폭이 매번 다르게 추정**되는 문제 발생.

## 6. 걸음 감지(Step Detection) 알고리즘 분석

### 6-1. 상태 기계 (State Machine) 개요

```
   ┌─────────────────────────────────────────────────────┐
   │                    IDLE STATE                       │
   │  isUpPeak=false, isDownPeak=false, isStepFinished=false  │
   └────────────────────┬────────────────────────────────┘
                        │ transformedZ > dynamicUpThresh
                        │ && 이후 하락 감지 (Z < maxAccZ)
                        ▼
   ┌─────────────────────────────────────────────────────┐
   │                  UP-PEAK DETECTED                   │
   │  isUpPeak=true, upPeakTime 기록, maxAccZ 확정       │
   └────────────────────┬────────────────────────────────┘
                        │ transformedZ < dynamicDownThresh
                        │ && 이후 상승 감지 (Z > minAccZ)
                        ▼
   ┌─────────────────────────────────────────────────────┐
   │                 DOWN-PEAK DETECTED                  │
   │  isDownPeak=true, downPeakTime 기록, minAccZ 확정    │
   └────────────────────┬────────────────────────────────┘
                        │ transformedZ >= 0.0 (영점 복귀)
                        │ && (currentTime - upPeakTime) > 350ms
                        ▼
   ┌─────────────────────────────────────────────────────┐
   │                 STEP FINISHED                       │
   │  isStepFinished=true → 유효성 검사 진입             │
   └────────────────────┬────────────────────────────────┘
                        │ 4가지 유효성 검사 통과
                        ▼
   ┌─────────────────────────────────────────────────────┐
   │               VALID STEP → return true              │
   │  totalStepCount++, ampWindow 강화, reset()          │
   └─────────────────────────────────────────────────────┘
```

**타임아웃**: up-peak 이후 2500ms 경과 시 자동 리셋 (미완성 걸음 폐기)

### 6-2. 적응형 임계값 시스템

가속도 데이터를 25-sample 윈도우(`ampWindow`)에 축적하여 **보행 강도에 따라 임계값을 자동 조절**:

```
effectiveRange = min(전체 윈도우 range, 최근 10샘플 range)
dynamicZDiff = max(바닥값, effectiveRange × 0.45)
dynamicUpThresh = dynamicZDiff × 0.4
dynamicDownThresh = -(dynamicZDiff × 0.4)
```

**3단계 전략**:

| 단계 | 조건 | dynamicZDiff | Up/Down Thresh | 목적 |
|------|------|-------------|----------------|------|
| 초기 | totalStepCount == 0 | **1.5** | ±0.6 | 손 떨림/기기 조작 노이즈 방어 |
| 전환 | totalStepCount < 3 | **1.0** | ±0.4 | 총총걸음 감지 가능하게 완화 |
| 적응 | totalStepCount ≥ 3 | **max(1.0, range×0.45)** | ×0.4 | 사용자 보행 강도에 자동 적응 |

**정지 감지**: `ampWindow`의 표준편차 < 0.25 → 정지 상태로 판정, 걸음 감지 중단

### 6-3. 유효성 검사 (4중 게이트)

걸음 완료 후 아래 4가지를 **모두** 통과해야 유효한 걸음으로 인정:

| 검사 | 조건 | 정상값 | 재시작 걸음 |
|------|------|--------|------------|
| **Peak-to-Peak 시간** | 200~2000ms | ✓ | 100~1500ms (완화) |
| **진폭** | maxAccZ - minAccZ ≥ dynamicZDiff | ≥1.0~1.5 | 동일 |
| **걸음 간격** | ≥ 250ms (직전 걸음 이후) | ✓ | 면제 |
| **Peak→Valley 시간** | ≥ 80ms | ✓ | 동일 |

**재시작 걸음(Restart Step)**: 첫 걸음 또는 2초 이상 정지 후 재개 시, peak-to-peak 대신 peak-to-valley 시간을 사용하고 조건을 완화.

### 6-4. 시간 가드 (350ms)

걸음 완료 판정에 고정 350ms 시간 가드 적용:

```cpp
else if (transformedZ >= 0.0) {
    bool isRealEnding = ((currentTime - upPeakTime) > 350.0);
    if (isRealEnding) {
        isStepFinished = true;
    }
}
```

- **목적**: 하강 영점 통과(~170ms)를 걸음 완료로 오인하는 것을 방지
- **동작**: up-peak 이후 최소 350ms가 지나야 영점 복귀(Z ≥ 0.0)를 완료로 인정
- **안전성**: 1.5~2.0Hz 보행(주기 500~667ms)에서 하강 영점(~170ms)은 확실히 차단되고, 상승 영점(~350-500ms)은 통과

| 보민 | 일반 | 539ms | 0.024 | 0.023 | 1.80~1.93Hz |
| 보민 | 느린 | 666ms | 0.041 | 0.040 | 1.39~1.59Hz |

**해석**:
- 일반 보행의 Duration CV는 0.024~0.078 범위 → **걸음 간격이 매우 일정**
- P5~P95 범위가 대부분 ±0.15Hz 이내 → **90%의 걸음이 좁은 주파수 대역에 집중**
- 느린 보행도 CV < 0.05로 안정적 (명권 느린 0.049, 보민 느린 0.041)

## 8. 미해결 이슈 및 개선 과제

### 8-1. 가은 천천히 걷기 과검출 (124.2%)

| 항목 | 값 |
|------|-----|
| 기대 걸음수 | 120 |
| 실제 감지(New) | 149 (+24.2%) |
| 실제 감지(Old) | 136 (+13.3%) |
| Freq CV | 0.215 (전체 최고) |
| P5~P95 범위 | 1.02~1.83Hz (매우 넓음) |

**원인 분석**:
- 천천히 걷기에서 보행 주기가 ~735ms(1.36Hz)로 길어지면서 하강-상승 사이에 **2차 진동**이 발생
- 적응형 임계값이 약한 2차 진동까지 잡아 과검출
- Old도 136으로 13% 과검출이므로 **데이터 자체가 비정형적** 가능성

**제안 수정**: 적응형 시간 가드 도입 검토 (Section 8-3 참고)

### 8-2. 350ms 고정 시간 가드의 한계

현재 걸음 완료 판정에 고정 350ms 시간 가드를 사용:

```
isRealEnding = (currentTime - upPeakTime) > 350.0
```

이 값은 1.5~2.0Hz 일반 보행(주기 500~667ms)에 최적화되어 있으나:
- **매우 느린 보행** (< 1.3Hz, 주기 > 770ms): 가드가 상대적으로 짧아 2차 진동 통과 가능
- **매우 빠른 보행** (> 2.5Hz, 주기 < 400ms): 가드가 상대적으로 길어 지연 발생 가능

### 8-3. 제안: 적응형 시간 가드

```cpp
// 제안 코드 (미적용 - 추가 검증 필요)
double adaptiveGuard;
if (lastStepPeakToPeak_ > 300.0 && totalStepCount >= 2) {
    adaptiveGuard = std::clamp(lastStepPeakToPeak_ * 0.55, 250.0, 500.0);
} else {
    adaptiveGuard = 350.0;
}
bool isRealEnding = ((currentTime - upPeakTime) > adaptiveGuard);
```

**원리**: 관측된 보행 주기의 55%를 가드값으로 사용하여 보행 속도에 자동 적응

| 피험자 | 주기 | 현재 가드 | 제안 가드 |
|--------|------|----------|----------|
| 보민 일반 (1.86Hz) | 538ms | 350ms | 296ms |
| 도훈 일반 (1.85Hz) | 541ms | 350ms | 297ms |
| 명권 느린 (1.47Hz) | 680ms | 350ms | 374ms |
| 가은 천천히 (1.36Hz) | 735ms | 350ms | 404ms |

**현재 상태**: 시뮬레이터 테스트 결과 일반 보행에서 추가 개선 효과 없음 (이미 350ms로 충분). 천천히 걷기 과검출 문제 해결 여부는 추가 검증 필요.
